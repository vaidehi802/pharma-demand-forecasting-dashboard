# RxForecast: Pharma Sales Demand & Territory Optimization Dashboard

A commercial pharma analytics pipeline: **quarterly demand forecasting** by
drug/product line (ARIMA, Python) + **HCP under-prescriber segmentation**
(scikit-learn classifier), delivered as an interactive Streamlit dashboard.
Built in the style of a ZS Associates-type commercial analytics engagement.

## What's real vs. simulated (read this first)

| Component | Data source | Status |
|---|---|---|
| Demand forecasting (Tab 1) | Kaggle "Pharma Drug Sales" dataset (`salesmonthly.csv`, 2014–2019) | **Real, public data** |
| Physician segmentation (Tab 2) | `data/physician_data.csv`, generated by `src/generate_data.py` | **Simulated** — physician-level prescribing data is proprietary/HIPAA-adjacent and not publicly available. The simulation is built so the label is realistically driven by the same features (digital engagement, rep visits, formulary access) that matter in real commercial engagements. |

Being upfront about this distinction is itself part of the deliverable —
it's exactly the kind of data-provenance question a hiring manager or client
will ask.

## Methodology

### 1. Demand forecasting (ARIMA)
- 70 months of real, monthly sales volume across 8 drug categories (ATC
  codes — e.g. `R03` = asthma/COPD drugs, `N05B` = anxiolytics).
- Dropped the final month, which the raw daily file showed was only 8/30
  days recorded — a real data-quality check, not just a formality.
- Log1p-transformed each series to stabilize multiplicative seasonality,
  then used `pmdarima.auto_arima` (stepwise AIC search, monthly seasonality
  `m=12`) per product line.
- Validated on a 6-month holdout, then refit on full history to forecast
  the next quarter.

**Result: 18.0% average MAPE across all 8 product lines (≈82% forecast
accuracy)**, with the two most stable categories (analgesics `N02BA`,
anxiolytics `N05B`) forecasting under 10% MAPE.

### 2. Physician under-prescriber classification (ML)
- Simulated 1,500 physicians × 12 quarters (region, specialty, panel size,
  digital engagement, rep visit frequency, formulary access, peer
  influence).
- Label: a physician is an "under-prescriber" if actual scripts fall below
  75% of what their panel's epidemiology would predict.
- `GradientBoostingClassifier` (scikit-learn), split by physician ID
  (not by row) to avoid leakage across quarters.

**Result: ROC-AUC 0.86**, with digital engagement, rep visit frequency,
and formulary access — the three levers a commercial team can actually act
on — as the top predictive features. (Precision/recall at the default 0.5
threshold are conservative because under-prescribers are a rare class ~7%
of physicians; in practice, reps would target the top-N by risk score
rather than use a hard cutoff.)

### 3. Territory recommendation
Combines both models: regions with a higher concentration of high-risk
(under-prescribing) physicians are flagged for rep reallocation or targeted
formulary-access outreach next quarter.

## How to run locally

```bash
pip install -r requirements.txt

# 1. Regenerate the simulated physician dataset (optional — already included)
python3 src/generate_data.py

# 2. Run the forecasting pipeline
python3 src/forecasting.py

# 3. Train the physician classifier
python3 src/ml_physician_model.py

# 4. Launch the dashboard
streamlit run dashboard/app.py
```

## Cloud deployment (free-tier friendly)

### Option A — Azure App Service (Docker)
```bash
az group create --name rxforecast-rg --location eastus
az acr create --name rxforecastacr --resource-group rxforecast-rg --sku Basic
az acr build --registry rxforecastacr --image rxforecast:v1 .
az appservice plan create --name rxforecast-plan --resource-group rxforecast-rg --is-linux --sku F1
az webapp create --resource-group rxforecast-rg --plan rxforecast-plan \
  --name rxforecast-demo --deployment-container-image-name rxforecastacr.azurecr.io/rxforecast:v1
```

### Option B — AWS (S3 for data + Streamlit on a free-tier EC2/App Runner)
```bash
aws s3 mb s3://rxforecast-data-<yourname>
aws s3 cp data/ s3://rxforecast-data-<yourname>/data/ --recursive
aws s3 cp outputs/ s3://rxforecast-data-<yourname>/outputs/ --recursive
# Then run the Docker image on AWS App Runner (free-tier eligible) pointing
# at the same Dockerfile, or on an EC2 t2.micro instance.
```

### Option C — Streamlit Community Cloud (simplest, fully free)
Push this repo to GitHub and connect it at share.streamlit.io — no
infrastructure to manage, just point it at `dashboard/app.py`.

Any one of these is enough to make "cloud-deployed" a true, verifiable line
on a resume — the point is a real deployment artifact (Dockerfile +
working app), not necessarily 24/7 hosting.
